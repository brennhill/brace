BRACE Guide · Agent

Identity & attribution

The Agent concern is about knowing which agent did what. When an agent does something it shouldn't, can you tell which agent — running which build, on whose behalf — did it? Without that, every incident is unattributable, and you can neither contain it nor learn from it.

Agents commonly inherit a person's identity or a shared service account, so two different agent builds look identical in your logs. The Agent concern fixes that with a distinct identity per agent, a small set of required fields on every action, and a content hash that pins exactly which build ran. Identity is the recovery surface: it turns an incident from "something went wrong somewhere" into "this build, this instance, under this parent, did this." This guide covers BRACE's identity fields and parent-child tracking.
The five principles
  1. Give the agent its own identity, separate from the launching user
  2. Stamp the six required fields on every action
  3. Make the agent-type-id a content hash
  4. Track parent-child lineage with the parent-passed prompt
  5. Treat identity as the recovery surface, not paperwork
PRINCIPLE 1

Give the agent its own identity, separate from the launching user

An agent should act as itself, not as the person who launched it and not as a shared service account. Non-human identity is distinct from human identity.

Why it matters. If the agent borrows a human's credentials, two things break: you can't separate what the person did from what the agent did, and the agent inherits all of that person's access — usually far more than its task needs. A shared service account is the same problem at scale: every agent build looks the same in the identity provider's logs. Distinct identity is the precondition for both attribution and least privilege.

PRINCIPLE 2

Stamp the six required fields on every action

Every action an agent takes should carry six fields: accountable party, operational owner, tenant, agent-type-id, agent-instance-id, and trace context.

Why it matters. These six answer the questions you'll actually ask during an incident: who is responsible for this, who operates it, for which customer, which build, which specific run, and where in the call graph. Most systems capture one or two of these. The missing ones are exactly what leave an incident unattributable — you can see that something happened without being able to say which agent or run it belongs to.

PRINCIPLE 3

Make the agent-type-id a content hash

Derive the agent-type-id from the build itself — a hash over the container image, harness, system prompt, model, and configuration — not from a name someone assigns.

Why it matters. A label like support-agent-v2 is just a name, and names get reused for changed builds. A content hash changes whenever any input changes, so two different configurations are always distinguishable, and a silent change — a hot-patched prompt, a swapped model, an added tool — shows up as a new id in your logs instead of hiding under the old name. It's the same idea as a container digest, extended to everything that determines how the agent behaves.

PRINCIPLE 4

Track parent-child lineage with the parent-passed prompt

When an agent spawns a sub-agent, record which sub-agent ran, under which parent, and the prompt the parent handed it.

Why it matters. In a multi-agent system, an action three levels deep means little without the chain above it. The parent-passed prompt is the instruction that actually drove the sub-agent — and it's often the only place an injected instruction is visible, since it may never appear in the user's input or the sub-agent's own configuration. Capturing the lineage and that prompt is what lets you trace a deep action back to its real cause.

PRINCIPLE 5

Treat identity as the recovery surface, not paperwork

Identity is not a compliance checkbox. It is the surface every response depends on, so build it in before you need it.

Why it matters. Containment, the kill switch, the audit trail, and learning from an incident all rest on attribution: you can't stop or explain what you can't identify. Identity added after an incident does nothing for that incident — the actions are already in the logs, unattributed. Treating it as foundational, the way you'd treat authentication for a normal service, is what makes the rest of BRACE work when it counts.

✗ Bolted on later
Identity is an afterthought; when something goes wrong, the logs show actions no one can tie to a specific build, run, or owner.
✓ Built in first
Every action is attributable from day one, so containment and audit have something to act on the moment they're needed.

Worked example — a fleet of per-customer support agents

One agent build runs as many instances, one per customer, some spawning sub-agents. Identity applied.

  • Own identity (P1): the fleet runs under its own non-human identity, not a shared service account or a support rep's login.
  • Six fields (P2): every action records accountable party, owner, tenant, type, instance, and trace context.
  • Content hash (P3): the agent-type-id is a hash of the build; a prompt hot-patch to one region shows up immediately as a different id.
  • Lineage (P4): a sub-agent that escalates a ticket carries its parent's id and the prompt the parent passed it.
  • Recovery surface (P5): all of this exists before launch, so an incident is attributable on the first day, not the next quarter.

When one customer reports something strange, you can name the exact build, instance, and parent involved — and tell whether every other customer's agent is the same build or a drifted one.

Agent go / no-go

A short gate before you ship. The full version lives in the BRACE sign-off checklist.

  • The agent has its own identity, separate from the launching user and from shared service accounts.
  • Every action carries the six fields: accountable party, operational owner, tenant, agent-type-id, agent-instance-id, trace context.
  • The agent-type-id is a content hash over container, harness, system prompt, model, and config.
  • Parent-child lineage and the parent-passed prompt are captured for every sub-agent.
  • Identity exists before launch — it isn't something you add after an incident.
