BRACE Guide · Ecosystem

Multi-agent & supply chain

An agent rarely works alone. It loads third-party tools, calls MCP servers, spawns sub-agents, and shares a substrate — a registry, an identity plane, a gateway, a kill-switch bus — with every other agent you run. Your agent is only as safe as the agents and tools it calls. Ecosystem security is about the parts you don't build but still depend on.

Every BRACE control has two halves: an agent-scoped half you configure per agent, and an ecosystem-scoped half that lives in the shared substrate. A well-built agent is still exposed if it calls a poisoned tool, or if it runs where the kill switch can't reach its sub-agents. This guide covers the six principles behind BRACE's ecosystem controls — supply chain, agent-to-agent boundaries, the MCP gateway, and recursive closure — with the reasons they matter and worked examples.
The six principles
  1. Your agent is only as safe as the agents and tools it calls
  2. Vet what you load — and keep vetting it
  3. Treat tool descriptions and schemas as untrusted input
  4. Watch for lookalikes
  5. Put a fail-closed gateway at the MCP boundary
  6. The kill switch must reach the sub-agents
PRINCIPLE 1

Your agent is only as safe as the agents and tools it calls

Security ends not at your agent's boundary but at the boundary of everything it touches — every tool, every sub-agent, and the shared substrate underneath them all. Map that whole surface, because that is your real attack surface.

Why it matters. The agent you wrote may be flawless, but the moment it loads a third-party MCP server, calls a sub-agent, or relies on the shared registry and identity plane, it inherits the weaknesses of all of them. This is also where failures cascade: one confused or hijacked agent spawns sub-agents, calls tools, and passes tainted output along, so a single bad node becomes a chain. The shared substrate (registry, IAM, MCP gateway, kill-switch bus, audit pipeline) is half of every control. Because it is owned by no single agent, it often gets neglected.

PRINCIPLE 2

Vet what you load — and keep vetting it

Vet every MCP server and tool before you load it; pin and sign it; fold its identity — a hash of its description and schema — into your records. Then re-check that hash on every load, because a tool that was safe at vetting time can change after approval.

Why it matters. A one-time review is not enough. The dangerous case is the rug pull: a tool you vetted and trusted ships a later version that quietly adds an exfiltration step or a hidden instruction. Publicly reported incidents include a popular MCP server whose npm package turned malicious in an update, and a remote-MCP client carrying a remote-code-execution flaw. You hash your own build into the agent-type-id so drift is visible. Do the same for your dependencies: fingerprint them and alert when the fingerprint moves.

✗ Vet once
Approve an MCP server at integration time, then auto-update it forever. The version you reviewed is not the version running.
✓ Vet, pin, re-check
Pin the tool by version/digest, store its fingerprint, and compare on every load. A changed fingerprint blocks the tool until a human re-vets it.
PRINCIPLE 3

Treat tool descriptions and schemas as untrusted input

A tool's description and parameter schema are injected into the agent's prompt — so they are an instruction channel, and an attacker controls them. Scan and sanitize them before they ever reach the model.

Why it matters. This is “tool poisoning.” A hidden instruction buried in a tool description runs with the agent's full authority, and the description is exactly the kind of text a model treats as guidance. Because it can be invisible in any UI you would review by eye, you can't catch it by reading. Scan for the specific vectors and strip what doesn't belong:

What to scan for: invisible Unicode and zero-width characters · right-to-left overrides · HTML/XML comments · base64/hex-encoded payloads · role-override patterns (“ignore previous…”, “you are now…”) · exfiltration URLs. Then: strip control characters and bound description length on registration.

PRINCIPLE 4

Watch for lookalikes

A malicious server can register a tool named almost exactly like a legitimate one, hoping the agent (or a careless config) calls the wrong one. Detect near-duplicate tool names across servers, and require explicit, namespaced tool identity.

Why it matters. This is the agent-ecosystem version of typosquatting. If payments.refund and payment.refund both resolve, the agent can be steered to the attacker's copy in a cross-server confused-deputy attack. Flag tool names within a small edit distance of ones you already trust on other servers, and namespace tools by a verified server identity so “which refund” is never ambiguous.

✗ Ambiguous
Tools resolved by bare name across all connected servers; send_email from any server is as good as any other.
✓ Namespaced + checked
Tools resolved by verified-server-id / tool; names within edit-distance 2 of a trusted tool on another server are flagged before use.
PRINCIPLE 5

Put a fail-closed gateway at the MCP boundary

Interpose a gateway between the agent and its tool servers that enforces allow/deny, scans tool calls and responses, authenticates and rate-limits, and audits every decision — and that fails closed: on any error, it denies, never silently permits.

Why it matters. Principles 2–4 are enforcement points; a gateway is where you enforce them consistently for every agent instead of re-implementing them per agent. It is the network-firewall pattern applied to MCP traffic: a single, auditable choke point. The “fail closed” rule is what makes the whole thing trustworthy. A scanner that errors open is worse than no scanner, because it gives false confidence. If a check can't run, the safe answer is to deny.

Reference implementation. Microsoft's open MCP Security Gateway 1.0 (part of the MIT-licensed Agent Governance Toolkit) is a conformance-tested spec for exactly this gateway — tool-call interception with approval workflows, response scanning (block/sanitize/log), a security scanner for the six threats above (tool poisoning, rug pull, cross-server attack, confused deputy, hidden instruction, description injection), schema-drift detection, and fail-closed semantics throughout. BRACE defines what to enforce at the MCP boundary and in what order; that spec is one concrete way to build it.

PRINCIPLE 6

The kill switch must reach the sub-agents

In a multi-agent system, halting the agent you launched is not enough. The stop signal must propagate recursively to every sub-agent it spawned and every tool session it opened — and leave work in a safe state. And it must be tested.

Why it matters. A confused or hijacked parent agent can spawn a tree of sub-agents that keep running after you “stopped” it. This is the multi-agent cascade. A kill switch that only halts the parent gives false assurance while the children continue to act. Recursive propagation needs the agent-to-agent boundary to carry identity and a stop channel, so the substrate can find and halt every descendant. A kill switch you have not tested is an assumption rather than a control, so exercise it. For designing the stop mechanism and its human controls, see LoopRails on kill switches.

✗ Halt the parent
The stop button kills the agent you launched; its sub-agents and open tool sessions keep running, unattributed.
✓ Recursive & tested
The stop signal propagates along the agent-to-agent graph to every descendant, each ending in a safe state — and you've tested that it does, on a schedule.

Worked example — an agent that calls a third-party MCP tool

One ordinary integration, where your agent uses a third-party MCP server for currency conversion, with all six principles applied.

  • Surface (P1): the tool, its server, and the shared gateway/registry are all in scope for review, not just your agent.
  • Vet & re-vet (P2): the server is pinned by version, fingerprinted at vetting, and re-checked on every load; a changed fingerprint blocks it.
  • Untrusted descriptions (P3): its tool descriptions are scanned for hidden instructions and stripped of control characters before reaching the model.
  • Lookalikes (P4): the tool is namespaced to its verified server; a near-duplicate convert on another server is flagged.
  • Gateway (P5): all of the above runs in a fail-closed MCP gateway that also rate-limits and audits every call.
  • Closure (P6): if you halt the agent, the gateway tears down its tool sessions too.

A malicious update to that third-party tool can't silently reach the model, can't run hidden instructions, and can't outlive a stop, even though you didn't write a line of it.

Ecosystem go / no-go

A short gate before you ship. The full version lives in the BRACE sign-off checklist.

  • Every third-party tool and MCP server is vetted, pinned, and signed — and re-checked (fingerprinted) on every load.
  • Tool descriptions and schemas are scanned for hidden instructions and sanitized before reaching the model.
  • Tools are namespaced to a verified server identity; lookalike names are flagged.
  • The MCP boundary runs through a gateway that scans, authenticates, rate-limits, audits — and fails closed.
  • The kill switch propagates recursively to every sub-agent and tool session, and has been tested.
  • Agent-to-agent calls carry identity and a stop channel across the boundary.
