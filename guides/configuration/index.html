BRACE Guide · Configuration

The agent is a configuration

There's no pull request to review for a running agent. The agent isn't the code you shipped. It's a configuration of infrastructure: a container, a harness, a system prompt, a set of tools, memory, an identity, a network path. Two agents built from the same model behave very differently depending on those parts. So the thing you review and version is not the code; it's the configuration.

Because the security-relevant artifact is the configuration, the disciplines you already use for code — version control, diff review, change approval, drift detection — have to move to the configuration layer. A one-line change to a system prompt can remove a guardrail with no pull request, no review, and no record. This guide covers the Configuration concern: the harness and system prompt as reviewed artifacts, the configuration frozen and pinned into identity, and drift detection at agent granularity.
The five principles
  1. The agent is a configuration, not the code that shipped
  2. Treat the system prompt and harness as versioned, diff-reviewed artifacts
  3. Freeze the configuration per release and pin it into identity
  4. Detect drift at agent granularity
  5. Capture the config-adjacent observables
PRINCIPLE 1

The agent is a configuration, not the code that shipped

The security-relevant artifact is the running configuration — container, harness, system prompt, tools, memory, identity, network — not the application code. Secure the configuration, because that's what determines behavior.

Why it matters. Take one model and wrap it two different ways, with a different system prompt, different tools, and different limits, and you get two agents that behave nothing alike. No code diff captures that difference, so a security review aimed at the code misses the real attack surface. Once you see the agent as a configuration, it is clear what to protect: the parts that decide what the agent is and does.

PRINCIPLE 2

Treat the system prompt and harness as versioned, diff-reviewed artifacts

Put the system prompt and harness in version control and change them only through review — the same discipline you apply to code. They should not be editable in production by anyone who feels like it.

Why it matters. The system prompt is policy. An editable-in-production prompt is an unreviewed, unattributed code path with full influence over the agent, and a one-line change can remove a guardrail or redirect the agent's goals. Several real incidents come down to exactly this: a prompt or rules file treated as runtime configuration that anyone could edit, with no diff and no history. Reviewing changes the way you review code is what makes the rest of your controls trustworthy.

PRINCIPLE 3

Freeze the configuration per release and pin it into identity

Freeze the full configuration for a release so "what's running" is a known, reproducible thing — and pin its hash into the agent's identity, as the agent-type-id.

Why it matters. Freezing turns the configuration from a moving target into a fixed, reproducible reference. Pinning that frozen configuration's hash into identity (see the Agent guide) makes any later change visible in your logs as a new id rather than a silent edit. Together they let you state plainly what configuration produced a given behavior — the precondition for both drift detection and attribution.

✗ Floating config
The prompt, tool list, and image tags can each change independently between deploys; "what's running" is whatever was last touched.
✓ Frozen & pinned
A release freezes the whole configuration; its hash is the agent-type-id, so any change is a new, visible build.
PRINCIPLE 4

Detect drift at agent granularity

Watch for drift on the things that define an agent build — container images, harness configs, MCP server lists, system prompts, and per-agent network policies — not just at the host or cluster level.

Why it matters. Configuration drifts: someone hot-patches a prompt, a tool list gains a server, an egress rule loosens "just for debugging." Infrastructure-as-code drift detection is a well-established discipline that compares running state against a declared baseline, but it is usually applied at host granularity, which is exactly where these agent-level changes hide. Applying the same comparison per agent build is what surfaces them.

PRINCIPLE 5

Capture the config-adjacent observables

The record of what configuration actually ran is incomplete without two more things: the decision-time context size and the prompt a parent passed to a sub-agent.

Why it matters. The configuration that produced a behavior is more than the static files. How full the context was when a decision was made, and what a parent told a sub-agent at run time, both shape what happened — and neither is in the frozen artifacts. Capturing them (covered in the Run-time and Agent guides) completes the picture, so "what ran" accounts for the dynamic inputs as well as the pinned ones.

✗ Static only
You can reproduce the files but not the run — no record of context size at each decision or the prompt a parent handed down.
✓ Full picture
Decision-time context size and parent-passed prompts are logged alongside the pinned configuration.

Worked example — a coding agent whose behavior is its config

An internal coding agent with a fixed model, where behavior is entirely a function of its harness, prompt, rules files, and tools. Configuration applied.

  • Config as the artifact (P1): the security review targets the harness, system prompt, rules files, MCP config, and tool list — not the application code.
  • Reviewed artifacts (P2): the system prompt and rules files live in version control and change only through reviewed pull requests.
  • Frozen & pinned (P3): each release freezes the whole configuration; its hash is the agent-type-id.
  • Drift detection (P4): a check compares the running prompt, MCP list, and egress policy against the pinned baseline and alerts on any difference.
  • Observables (P5): decision-time context size and parent-passed prompts are logged so a run can be reconstructed, not just the files.

A prompt edited directly in production shows up as drift against the pinned baseline within minutes, instead of quietly changing how every developer's agent behaves.

Configuration go / no-go

A short gate before you ship. The full version lives in the BRACE sign-off checklist.

  • The security review targets the configuration — container, harness, prompt, tools, memory, identity, network — not just the code.
  • The system prompt and harness are in version control and change only through review.
  • Each release freezes the full configuration, and its hash is pinned as the agent-type-id.
  • Drift detection runs at agent granularity: images, harness, MCP lists, prompts, and per-agent egress.
  • Decision-time context size and parent-passed prompts are logged alongside the pinned configuration.
