BRACE Guide · Build-time

Building agents correctly

Build-time is everything you decide and freeze before the agent ever runs: its tools, credentials, network paths, container, and harness rules. It is the cheapest and most reliable place to make an agent safe, because these controls don't depend on the model behaving. Get them right and most catastrophic outcomes become impossible by construction rather than merely discouraged.

A confused, jailbroken, or prompt-injected agent still cannot do what its build never gave it the means to do. Run-time controls detect and respond; build-time controls bound what is possible at all. This guide covers the six principles behind BRACE's build-time controls — Architecture, Capability-scoped access, Container, and Harness — with the reasons they matter and worked examples.
The six principles
  1. Give the agent the least it needs — and decide it in advance
  2. Bound the blast radius by topology, not by trust
  3. Scope and time-box every credential
  4. Gate destructive actions in the harness, deny-by-default
  5. Pin and sign the build; make it part of the agent's identity
  6. Treat the harness and system prompt as reviewed artifacts
PRINCIPLE 1

Give the agent the least it needs — and decide it in advance

An agent should have access to nothing it doesn't need, and every access should be granted ahead of time — not discovered, escalated, or improvised at run-time. This applies equally to tools, MCP servers, credentials, file paths, and network destinations.

Why it matters — and the two reasons point the same way:

Security: scope is your blast radius. Whatever you grant is exactly what a hijacked agent can do. Prompt injection, a poisoned web page, or a confused plan can't cause harm if the capability was never there. Least privilege changes the requirement from “the model must always behave” to “the model can't do harm even when it misbehaves.”

Capability: fewer tools work better. Teams often get this backwards. Every tool and every MCP server you attach is injected into the model's context as a schema it must read on every step. Twenty tools can be thousands of tokens of menus the model reads before it starts your task, which is context budget spent on options instead of reasoning. More choices also degrade selection: the model picks the wrong tool, invents arguments, or chains tools that shouldn't be chained. A focused toolset is safer, and it is also faster, cheaper, and more accurate. Adding a tool “just in case” carries a real, recurring cost on every run.

Minimal scope is one of the few controls that improves security and performance at the same time.

✗ Over-scoped
A ticket-triage agent given a full GitHub MCP, a filesystem tool, the AWS CLI, a shell, and 14 more tools “so it's flexible.” This is token-heavy, error-prone, and one injection away from touching prod. tools: [github(*), shell, fs, awscli, ...14 more]
✓ Right-scoped
The same agent given exactly three tools, chosen in advance. Lean context, reliable selection, and the worst it can do is post a bad comment. tools: [read:tickets, post_comment, search_kb]

Rule of thumb: if you can't name why a tool is there and what it's allowed to touch, remove it. Start from zero and add deliberately — don't start from “everything” and prune.

PRINCIPLE 2

Bound the blast radius by topology, not by trust

Put the agent where it physically cannot reach what it must never touch. Network and environment isolation is the control that holds even when every other control fails — because it doesn't rely on the agent deciding anything.

Why it matters. An agent can't delete a production database it has no network route to. It can't exfiltrate to an endpoint that egress rules block. Topology does not depend on your threat model: it contains mistakes, jailbreaks, and attacks you haven't imagined, because the path simply isn't there. Several publicly reported incidents where an agent destroyed a production database during a freeze share one root cause: the agent's environment had a route to prod and credentials for it. The fix is not a better prompt. It is removing the route.

✗ Trust-based
Agent runs in an environment with VPC access to prod and a broad egress: allow-all. Safety depends on the agent choosing not to misuse it.
✓ Topology-based
Agent runs in an isolated segment; egress allowed only to the two services it needs. Prod, payments, and admin are unreachable regardless of what the agent tries.
PRINCIPLE 3

Scope and time-box every credential

Tokens grant specific capabilities, never wildcards, and they expire. No standing god-credentials live in the agent's environment.

Why it matters. Topology limits where the agent can go; credentials limit what it can do once there. A read:tickets token can't authorize a deletion even if the agent is tricked into trying. Short lifetimes shrink the window a leaked or misused credential is good for, and narrow scopes make the audit trail meaningful — you can tell what an action could possibly have been. Wildcard tokens turn one mistake into a total compromise.

✗ Standing & broad
A long-lived personal access token, or cloud keys with full reach. scopes: [repo, admin:org] aws: { "Action": "*", "Resource": "*" }
✓ Scoped & short-lived
A per-run token, minimum scopes, auto-expiring. scopes: [read:tickets, write:comments] ttl: 15m issued: per-run
PRINCIPLE 4

Gate destructive actions in the harness, deny-by-default

The harness — not the model — decides whether an irreversible action is allowed. Keep a deny-by-default list for destructive verbs (delete, DROP, force-push, infra teardown, mass external send); high-risk actions require a second factor such as human approval or an out-of-band confirmation.

Why it matters. This is the highest-leverage build-time control. The model's judgment is not a safety boundary: it can be wrong, jailbroken, or injected. A gate in the harness is deterministic. It stops the most common catastrophic incidents (dropped tables, rm -rf, force-pushes, mass deletes) regardless of why the agent tried. You don't enumerate every safe action; you block a short list of irreversible ones and route them through a human. For the human side of that gate — deciding which actions actually need review, and keeping approvals from becoming rubber stamps — see the LoopRails human-in-the-loop framework; on the implementation side, n8n ships a human-in-the-loop node for AI tool calls.

✗ Trusting the model
The agent has a shell and “knows not to” run destructive commands. One injection or bad plan removes that assumption.
✓ Gating in the harness
The harness intercepts every call. Read and comment pass; DROP TABLE, rm -rf, and git push --force are denied or sent to a human approval step.
PRINCIPLE 5

Pin and sign the build; make it part of the agent's identity

Ship a minimal, signed container pinned by digest, and record that digest — along with the harness, system prompt, model, and config — in the agent's identity. A silently changed build should be detectable.

Why it matters. The agent's behavior is determined by its build. If the build can change without anyone noticing, so can its behavior, including maliciously through a poisoned dependency or a tampered image. A pinned, signed image gives you integrity and reproducibility, and a minimal image removes tools an attacker could pivot to (no shell, no curl if you don't need them). The supply-chain failure mode is real: a destructive change merged into a widely distributed developer extension shipped to roughly a million users. Folding the digest into the content-hashed agent-type-id means drift shows up in your logs instead of staying invisible.

✗ Mutable
image: agent:latest A full base image with a shell and package manager; “latest” can change underneath you.
✓ Pinned & signed
image: agent@sha256:9f2b… base: distroless · cosign: verified Digest folded into agent-type-id, so drift is visible.
PRINCIPLE 6

Treat the harness and system prompt as reviewed artifacts

The harness and system prompt are security-relevant code. Version them, diff-review them in pull requests, and freeze them per release. They should not be editable in production by anyone who feels like it.

Why it matters. Two agents with the same model behave very differently depending on their prompt and harness. So an editable-in-prod system prompt is an unreviewed, unattributed code path with full influence over the agent. A one-line prompt change can remove a guardrail or redirect the agent's goals. Applying the same change-review discipline you use for code to these artifacts is what makes the rest of your controls trustworthy. (This is where Build-time meets Configuration; see that guide.)

✗ Editable config
The system prompt lives in a database row an ops dashboard can edit live — no review, no history, no attribution.
✓ Reviewed artifact
Prompt and harness config live in version control, change only via reviewed pull requests, and are hash-pinned at deploy.

Worked example — a support-ticket triage agent, built right

The six principles aren't separate checkboxes; they compose. Here is one ordinary agent with all six applied.

  • Scope (P1): three tools — read:tickets, post_comment, search_kb. No shell, no filesystem, no extra MCP servers.
  • Topology (P2): an isolated segment; egress only to the ticket API and the knowledge base. No route to prod, payments, or admin.
  • Credentials (P3): a short-lived token scoped to read:tickets write:comments, issued per run.
  • Gate (P4): the harness denies every destructive verb; the only write is a comment, and even that is rate-limited.
  • Build (P5): a distroless image, pinned by digest and signed; the digest is part of its agent-type-id.
  • Artifacts (P6): the prompt and harness live in git, reviewed by PR, frozen at the deployed digest.

In the worst case, fully hijacked, it posts a bad comment on a ticket. That is the entire blast radius.

Build-time go / no-go

A short gate before you ship. The full version lives in the BRACE sign-off checklist.

  • Every tool, MCP server, and credential is named and justified; defaults removed.
  • The agent has no network route to systems it must never touch.
  • All credentials are capability-scoped and time-boxed; no wildcards.
  • Destructive verbs are denied by default in the harness; high-risk actions need human approval.
  • The container is minimal, signed, and pinned by digest.
  • The harness and system prompt are in version control and change only by review.
